架构识别与分类

本项目采用典型的裸机系统（Bare-Metal）架构，使用经典的前后台系统设计模式。

💡 核心设计理念

中断服务程序（ISR）快速设置事件标志位，主循环（Super Loop）轮询检查标志并执行具体业务逻辑，实现前后台分离。

标志位与状态机

标志位类型分析

类型 标志位 特点 用途
事件标志 g_isStartSampling 单次触发 响应外部事件
周期标志 g_isSysTickInt 周期触发 定时任务调度
同步标志 g_isA_Done, g_isB_Done 多条件组合 任务同步屏障
⚠️ 标志位 vs 状态机

这些标志位不是完整的状态机，而是事件驱动标志。标志位无状态保持、转换隐式、不可追溯；而状态机有明确的状态记录、显式转换条件和历史可追溯性。

隐式状态转换流程

stateDiagram-v2 [*] --> IDLE IDLE --> WAITING: EXTI触发
g_isStartSampling=true WAITING --> SAMPLING: MeasureTask()
开始采样 SAMPLING --> DATA_READY: DMA完成
g_isA_Done && g_isB_Done DATA_READY --> ANALYZING: AnalyzeTask()
数据分析 ANALYZING --> OUTPUT: g_isVoutReady=true
输出就绪 OUTPUT --> IDLE: OutputTask()
完成输出

代码示例：快速中断响应

/*!
    \brief  SysTick中断处理
    \note   只设置标志位，避免长时间占用中断
*/
void SysTick_Handler(void)
{
    delay_decrement();        // ✅ 快速操作1: 毫秒延迟计数
    tick_count++;             // ✅ 快速操作2: 时间戳累加
    g_isSysTickInt = true;    // ✅ 快速操作3: 设置事件标志
    
    // ✅ 总耗时 < 1μs，快速返回
}

架构缺陷

🔒 优先级固化

  • 优先级由if顺序决定
  • 无法动态调整
  • 扩展性差

🚫 任务饥饿

  • 长任务阻塞后续
  • 无抢占机制
  • 可能导致超时

📉 时序不稳定

  • 循环周期不固定
  • 标志位可能覆盖
  • 事件可能丢失

⛔ 功能限制

  • 不支持阻塞等待
  • 同步复杂
  • 竞态风险高
🚨 关键问题：标志位覆盖

如果中断频繁触发，而主循环处理速度较慢，后续触发的事件会被忽略（标志位已经是true）。解决方案：使用计数器代替布尔标志。

// ❌ 问题代码：标志位可能丢失事件
void ISR(void) {
    g_flag = true;  // 如果已经是true，新事件丢失
}

// ✅ 改进方案：使用计数器
volatile uint32_t g_eventCount = 0;

void ISR(void) {
    g_eventCount++;  // 计数不会丢失
}

void main_loop(void) {
    if(g_eventCount > 0) {
        uint32_t count = g_eventCount;
        g_eventCount = 0;
        
        for(uint32_t i = 0; i < count; i++) {
            ProcessEvent();
        }
    }
}

优化建议

短期优化（立即可行）

1️⃣ 防止标志位丢失

使用计数器代替布尔标志，避免高频事件丢失。

2️⃣ 任务超时检测

监控每个任务的执行时间，发现异常及时告警。

3️⃣ 看门狗保护

防止主循环卡死，添加硬件/软件看门狗。

4️⃣ 优化处理顺序

按任务紧急程度调整if语句检查顺序。

中期优化（需要重构）

// 引入简单调度表
typedef struct {
    TaskFunc func;
    volatile bool *flag;
    uint32_t priority;
    uint32_t budget_us;
} TaskEntry_t;

const TaskEntry_t g_taskTable[] = {
    {SysTickTask,   &g_isSysTickInt,     1, 50},
    {MeasureTask,   &g_isStartSampling,  2, 200},
    {AnalyzeTask,   &g_isAnalyzeReady,   3, 500},
    {OutputTask,    &g_isVoutReady,      4, 100},
};

void Scheduler_Run(void)
{
    // 按优先级顺序检查
    for(int i = 0; i < ARRAY_SIZE(g_taskTable); i++) {
        if(*g_taskTable[i].flag) {
            *g_taskTable[i].flag = false;
            Task_Execute(g_taskTable[i].func, i);
        }
    }
}

长期优化（架构升级）

  • 迁移到轻量级协作式内核
  • 选择性引入FreeRTOS特性
  • 混合架构：关键任务用RTOS，简单任务保持Super Loop

适用场景

✅ Super Loop 最适合

简单控制系统

  • 传感器数据采集
  • 电机PWM控制
  • 简单开关逻辑

资源受限设备

  • Flash < 32KB
  • RAM < 4KB
  • 低功耗要求

快速原型开发

  • 验证概念
  • 功能演示
  • 学习阶段

明确的任务结构

  • 任务数量 < 10
  • 依赖关系简单
  • 周期性任务为主

⚠️ 应该升级到 RTOS

以下情况建议使用 FreeRTOS：
  • 任务数量 > 10个
  • 硬实时要求（响应 < 1ms）
  • 需要阻塞操作（网络、文件系统）
  • 复杂的任务间通信
  • 长期维护的商业项目

📊 结论

当前 Super Loop 架构基本合适，建议：

  • 短期：优化标志位处理（防丢失）
  • 中期：引入简单调度表（提高可维护性）
  • 长期：如功能扩展（如加入通信、UI），考虑 FreeRTOS
